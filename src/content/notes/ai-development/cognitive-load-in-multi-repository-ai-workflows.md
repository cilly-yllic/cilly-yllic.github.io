AI コーディングアシスタントが「単一 repo の中」で動いている分には、認知負荷はあまり問題にならない。AI が文脈を読み、差分を出し、人間がレビューする。完結したループだ。

問題は repo が 2 つ以上に増えてからやってくる。

## どこが重いのか

たとえば、フロント repo と バックエンド repo を別々に持つ構成。フロントの新機能のために、バックエンドに API を 1 つ追加する。AI に依頼すると、

1. バックエンド repo を開いて API を実装させる
2. その output を見てフロント repo に戻り、クライアント側の使用箇所を AI に書かせる
3. その間に「バックエンドで決めたインタフェース」を人間が頭の中で保持する

この 3. が地味に重い。**AI のセッションが repo ごとに分かれている** と、人間がインタフェースの一貫性を保証する責任を負うことになる。これは型システムでも CI でもなく、人間の短期記憶が担う。

## モノリポはこれを解く

Nx / Turborepo のようなモノリポは、この問題に対して構造的に効く。AI セッションを 1 つに保てるからだ。API 変更とその呼び出し側の更新が、AI から見て同じ "プロジェクト" の中に閉じている。型が型のまま伝搬し、人間が頭の中で同期する必要がない。

「モノリポは AI に優しい」というのは、ツールチェインの話ではなく **コンテキスト境界の話** だ。

## それでも分けたいとき

repo を分けたい正当な理由はある (デプロイ単位、権限境界、技術スタックの違い)。その場合に効くのは、

- 共有 schema を **コミットしてもらう前の中間ステップとして AI に書かせる**
- フロントとバックエンドのインタフェースを **OpenAPI / GraphQL schema 等の機械可読な形** に固定する
- AI に「次の変更でこの schema を破ったら止める」と言わせる

要するに、AI 間の「会話」を人間ではなく schema に媒介させる、ということ。

## 経験則

repo を分割するときに払うコストは、CI / デプロイ複雑度よりも「人間の認知連結コスト」の方が大きい。これは AI を使い始めてから顕在化したコストで、AI 以前は「人間の暗黙知」として隠れていた。AI が当たり前になるほど、repo 分割の判断基準は変わる。
